# ADR-002: In-Memory Storage vs Database ашиглах шийдвэр

## Огноо
2026-05-10

## Статус
Баталгаажсан (Accepted)

## Контекст

Build хийх явцад өгөгдөл хадгалах давхаргыг хэрхэн зохион байгуулах талаар
шийдвэр гаргах шаардлага тулгарсан. Хоёр сонголт байсан:

1. **In-memory storage** — Java List ашиглан санах ойд хадгалах
2. **SQLite database** — файл дээр суурилсан жижиг database ашиглах

AI-тай ярилцсан үр дүнд дараах давуу болон сул талуудыг тодорхойлсон.

## Авч үзсэн сонголтууд

### Сонголт 1: In-Memory Storage (Java List)
**Давуу тал:**
- Хэрэгжүүлэхэд хялбар, нэмэлт dependency шаардахгүй
- Unit test бичихэд тохиромжтой, тусгаарлагдсан байна
- Maven тохиргоо энгийн

**Сул тал:**
- Программ дуусахад өгөгдөл устна
- Том өгөгдөлд тохиромжгүй

### Сонголт 2: SQLite + JDBC
**Давуу тал:**
- Өгөгдөл файлд хадгалагдана, программ дахин эхлэхэд хадгалагдана
- Бодит production системд ойр

**Сул тал:**
- JDBC driver dependency нэмэх шаардлагатай
- SQL query бичих шаардлагатай, кодын хэмжээ нэмэгдэнэ
- Unit test бичихэд илүү төвөгтэй

## AI-тай ярилцлага (товч)

**Хэрэглэгч:** In-memory эсвэл SQLite аль нь энэ төсөлд тохиромжтой вэ?

**Claude:** Энэ бие даалтын зорилго нь AI workflow эзэмших учраас
in-memory storage нь хангалттай. Repository pattern ашигласнаар
сүүлд database руу шилжих хялбар байна — зөвхөн Repository
layer-ийг солих хэрэгтэй болно.

## Шийдвэр

**In-memory storage (Java List) сонгосон.**

Repository pattern ашигласнаар өгөгдөл хадгалах давхарга
service layer-аас тусгаарлагдсан байна. Ирээдүйд SQLite эсвэл
өөр database руу шилжихэд зөвхөн Repository layer-ийг солиход хангалттай.

## Үр дагавар

- Unit test бичих хялбар болсон — тест бүр тусгаарлагдсан өгөгдөлтэй
- Программ дахин эхлэхэд өгөгдөл устдаг — demo зориулалтад хангалттай
- Repository pattern нь ирээдүйн өргөтгөлд бэлэн байна
